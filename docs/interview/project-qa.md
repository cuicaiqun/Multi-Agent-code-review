# 项目常见问题回答

## 一、项目背景类

### Q: 这个项目的业务背景是什么？
法律科技(LegalTech)是AI应用的热门领域。企业法务部门每天处理大量合同，人工审查耗时且容易遗漏。这个系统用多Agent架构自动化合同审查流程，将审查时间从45分钟/份压缩到8分钟/份。

### Q: 为什么要做三种语言版本？
1. **Python**：AI/ML领域的事实标准，LangGraph生态最成熟
2. **Java**：企业级后端的主流选择，Spring AI快速发展中
3. **Go**：字节跳动等公司的技术栈，Eino框架是新兴力量

不同语言覆盖不同的面试需求，也展示了同一架构在不同技术栈下的实现差异。

### Q: 项目中你具体负责什么？
我是核心开发者，负责：
1. 整体架构设计（四Agent流水线 + 三层校验）
2. Python版本的完整实现（LangGraph编排 + 四个Agent + 规则引擎 + NLP模块）
3. API接口设计和前端对接
4. 人机协同机制设计

## 二、技术选型类

### Q: 为什么选LangGraph而不是CrewAI或AutoGen？

| 维度 | LangGraph | CrewAI | AutoGen |
|------|----------|--------|---------|
| 控制精度 | 最高（显式图） | 中等（角色委托） | 低（对话循环） |
| 人机协同 | 原生interrupt | 不支持 | 有限支持 |
| 状态管理 | checkpoint持久化 | 无 | 消息历史 |
| 生产稳定性 | 高（LangChain团队） | 中 | 中 |
| 学习曲线 | 较陡 | 平缓 | 中等 |

**选择LangGraph的核心原因**：合同审查是法律场景，需要精确控制每一步的执行逻辑，而且人机协同是刚需。

### Q: 为什么用MiniMax而不是OpenAI？
1. MiniMax M2.7是国产大模型中性价比很高的选择
2. 支持中文法律文本理解
3. 通过OpenAI兼容接口调用，代码无需修改就可以切换到其他模型
4. 成本更低，50TPS的吞吐量满足需求

### Q: 如果不用大模型，纯规则引擎行不行？
行，但效果差很多：
- 纯规则方案：准确率约62%，只能检测已知的风险模式
- 规则+大模型：准确率94%，能发现语义层面的风险
- 大模型的价值在于：理解上下文、发现隐含风险、生成自然语言建议

## 三、实现细节类

### Q: 条款提取Agent是怎么工作的？

```
输入: 合同文档(PDF/DOCX/TXT)
     ↓
步骤1: 文档解析 (PyPDF2/python-docx) → 纯文本
     ↓
步骤2: 条款分割 (正则匹配"第X条"格式)
     ↓
步骤3: 调用大模型提取结构化信息
       - 条款分类 (payment/liability/...)
       - 实体识别 (甲乙方/金额/日期)
     ↓
输出: 结构化条款列表 [Clause对象]
```

**回退机制**：LLM调用失败时，使用纯正则表达式切分条款，保证系统可用。

### Q: 风险识别的双引擎架构怎么工作？

```
条款列表
  ├── 规则引擎 (确定性检查)
  │   ├── 关键词匹配: "无限责任"→高风险
  │   ├── 关键词匹配: "单方解除"→高风险
  │   └── ...
  │
  ├── 大模型分析 (语义理解)
  │   ├── 理解上下文含义
  │   ├── 买卖方偏向分析
  │   └── 隐含风险识别
  │
  └── 结果合并 (merge_findings)
      ├── 去重: 同一条款+同一风险类型
      └── 取最高等级: 两个引擎结果不同时取更高风险
```

### Q: 版本对比(diff)是怎么实现的？

使用Python的`difflib.unified_diff`生成条款级别的差异对比：
1. 对于每个修改建议，取原始条款文本和建议修改后的文本
2. 按行生成unified diff格式
3. 前端用红绿色展示增删改

**语义diff vs 文本diff**：
- 文本diff：字符级别的差异
- 语义diff：通过大模型理解修改的语义含义（如"将无限责任改为有限责任"）

## 四、性能与优化类

### Q: 系统的QPS是多少？瓶颈在哪？
- 规则引擎：单条款 < 5ms，QPS > 10000
- 大模型调用：2-5秒/次，是主要瓶颈
- 整个流水线：8-20秒/份合同（取决于条款数量）
- 并发处理：Python asyncio / Java CompletableFuture / Go goroutine

**优化方向**：
1. 风险识别和合规检查并行执行（无依赖关系）
2. 相似条款结果缓存（Redis）
3. 多份合同并行审查
4. MiniMax的Prompt Caching降低token成本

### Q: 如果大模型API挂了怎么办？
1. **重试机制**：指数退避重试3次
2. **熔断器**：连续失败5次触发熔断，60秒后半开
3. **降级方案**：切换到纯规则引擎模式
4. **多模型路由**：主模型不可用时切换备用模型
5. **用户提示**：告知用户当前为降级模式，结果可能不完整

## 五、扩展设计类

### Q: 如果要支持英文合同怎么改？
1. 文档解析器增加英文切分规则
2. NLP模块增加英文实体识别（人名、金额格式不同）
3. 规则引擎增加英文关键词库
4. Prompt翻译为英文版本
5. 大模型本身支持多语言，改动最小

### Q: 如果要支持10万份/天的审查量怎么设计？
1. **微服务拆分**：每个Agent独立部署为微服务
2. **消息队列**：Kafka/RabbitMQ异步解耦
3. **弹性伸缩**：K8s HPA根据队列长度自动扩容
4. **缓存层**：Redis缓存相似条款的分析结果
5. **批处理**：合并相似请求批量调用大模型
6. **读写分离**：审查结果写入异步，查询走缓存

### Q: 如何做Agent的持续优化？
1. **评估框架**：建立标准测试集（标注了正确答案的合同）
2. **A/B Testing**：新旧Prompt对比测试
3. **人工反馈闭环**：法务人员的修改意见用于优化Agent
4. **监控告警**：追踪每个Agent的准确率趋势
5. **Prompt版本管理**：Git管理Prompt，可回滚
