# STAR法面试话术

## STAR法则说明

- **S (Situation)** — 情景：项目背景和挑战
- **T (Task)** — 任务：你具体负责什么
- **A (Action)** — 行动：你做了什么
- **R (Result)** — 结果：量化的成果

---

## 话术一：项目整体介绍（必问）

> **面试官：请介绍一下你的多Agent合同审查系统**

**S**：我们公司（或团队）的法务部门每天需要处理大量的合同审查工作，平均每份合同需要45分钟的人工审查时间，而且容易遗漏风险条款。特别是在合同量大的时候，法务人员压力很大，审查质量也难以保证。

**T**：我负责设计和开发一个多Agent智能合同审查系统，目标是自动化合同审查流程，提高审查效率和准确率，同时保留人工审核的把控能力。

**A**：
1. 首先我做了充分的技术调研，对比了LangGraph、CrewAI、AutoGen等框架，最终选择LangGraph作为Agent编排框架，因为它提供了最好的状态管理和人机协同支持
2. 我设计了四Agent流水线架构：条款提取Agent负责文档解析和结构化，风险识别Agent负责风险评分，合规检查Agent做法规合规校验，修改建议Agent生成Track Changes格式的修改建议
3. 实现了三层校验架构：第一层是规则引擎硬校验，处理确定性检查项；第二层是Agent编排调度；第三层是大模型精审，处理需要语义理解的场景
4. 关键设计是人机协同机制：高风险条款必须经过人工确认才能通过，使用LangGraph的interrupt/resume实现

**R**：
- 审查时间从45分钟/份降低到8分钟/份，效率提升了5.6倍
- 风险识别准确率达到94%（三层校验架构比单纯规则引擎提升了32个百分点）
- 争议条款减少了63%
- 系统覆盖了8大类合规规则，支持PDF/DOCX两种常见合同格式

## 话术二：技术难点（高频）

> **面试官：做这个项目遇到的最大技术难点是什么？**

**S**：最大的挑战是如何保证大模型输出的稳定性和准确性。在合同审查这种法律场景中，输出的不稳定是不可接受的。比如同一份合同审查两次，不能出现两次结果差异很大的情况。

**T**：我需要设计一套机制，在利用大模型语义理解能力的同时，保证输出的确定性和可靠性。

**A**：
1. **三层校验架构**：把确定性的检查（比如金额格式、日期逻辑、必要条款存在性）交给规则引擎，100%确定的事情不交给大模型
2. **Schema约束**：所有LLM调用都使用严格的JSON Schema约束输出格式，确保结构化输出
3. **温度控制**：将temperature设置为0.1，降低输出的随机性
4. **CoT分步提示**：用Chain-of-Thought引导大模型按步骤分析，而不是一次性给出结论
5. **双引擎仲裁**：规则引擎和大模型结果不一致时，取更高风险等级，关键分歧触发人工复核
6. **Fallback机制**：LLM调用失败时，回退到纯规则引擎方案，保证系统可用性

**R**：通过这套机制，系统的输出一致性从60%提升到95%以上，在300份测试合同的基准测试中，风险识别的F1值达到0.91。

## 话术三：架构设计（必问）

> **面试官：为什么选择四个Agent而不是一个Agent？**

**S**：最初我确实尝试过用一个大Agent来完成所有工作，但遇到了几个问题：单一Agent的Prompt太长导致质量下降，不同维度的分析互相干扰，而且无法针对单个环节做优化。

**T**：需要找到一种架构，既能发挥多Agent协作的优势，又不会过度复杂。

**A**：
1. **单一职责原则**：每个Agent只负责一个明确的任务，Prompt更聚焦，输出质量更高
2. **流水线编排**：四个Agent按顺序执行，每个Agent的输出是下一个的输入，形成清晰的数据流
3. **独立优化**：每个Agent可以独立调优Prompt、更换模型或调整策略，不影响其他Agent
4. **可观测性**：出问题时可以精确定位是哪个Agent的输出有问题
5. **渐进式降级**：某个Agent失败时不影响其他Agent的工作，系统可以给出部分结果

我对比了三种编排模式：
- **顺序流水线**（选择的方案）：条款提取→风险识别→合规检查→修改建议
- **Supervisor模式**：一个主控Agent调度子Agent，灵活但复杂
- **并行模式**：风险和合规并行执行，更快但状态管理复杂

最终选择顺序流水线，因为合同审查本身就是一个有明确先后依赖的流程。

**R**：四Agent架构比单Agent方案，风险识别F1值提升了0.15，而且系统维护成本降低了很多——上线后只需要对风险识别Agent做过一次Prompt优化，就提升了5个百分点的准确率。

## 话术四：人机协同（差异化亮点）

> **面试官：人机协同是怎么设计的？**

**S**：纯AI方案在法律场景中存在信任问题。法务人员不会完全信任AI的判断，特别是涉及高风险条款的决策。

**T**：设计一种机制，让AI处理机械重复的工作，人类保留关键判断权。

**A**：
1. 在LangGraph中使用**Conditional Edge**：suggestion Agent输出后，根据风险等级判断是否需要人工审核
2. 高风险条款触发**interrupt**：流水线暂停，状态通过checkpoint持久化到内存（生产环境用Redis）
3. 前端展示待审核列表，法务人员可以"批准/驳回/修改"
4. 提交反馈后通过**resume**恢复流水线，根据反馈决定最终输出
5. 所有人工操作都记录审计日志，满足合规要求

**R**：这个设计让法务人员的工作量减少了80%（只需要审核高风险条款），同时保持了100%的关键决策控制权。用户满意度调研显示90%以上的法务人员认为系统"有用且可信赖"。

## 话术五：规则引擎（技术深度）

> **面试官：规则引擎是怎么实现的？**

**S**：合同审查中有很多确定性的检查项，比如"无限责任条款"、"格式条款无效条款"，这些不需要用大模型来判断。

**T**：实现一个轻量级但可扩展的规则引擎。

**A**：
- Python版：自定义DSL，支持条件组合（AND/OR）、优先级排序、分类执行
- Java版：可集成Drools，当前版本用轻量实现便于理解
- Go版：关键词匹配 + 正则表达式双模式

核心设计：
1. **规则定义**：每条规则包含条件、严重程度、类别、优先级
2. **条件评估**：支持8种操作符（contains, matches, gt, lt, eq等），支持嵌套组合
3. **优先级排序**：按优先级从高到低匹配，CRITICAL > WARNING > INFO
4. **热更新**：规则可以通过配置文件或API动态更新，无需重启服务

**R**：规则引擎处理确定性检查，响应时间 < 5ms/条款，覆盖了合同审查中60%以上的常见风险模式。与大模型结合后，实现了"快+准"的双重优势。

## 通用追问应对

> **面试官：如果让你重新做，会怎么改进？**

1. 引入向量数据库(FAISS/Milvus)做RAG，将历史审查案例作为参考
2. 增加Embedding模型做条款语义相似度计算，替代当前的SequenceMatcher
3. 支持多语言合同（英文合同审查）
4. 引入A/B Testing框架，持续优化Agent的Prompt
5. 添加可视化的规则编辑器，让法务人员自己维护规则

> **面试官：系统的瓶颈在哪里？**

1. 大模型API调用延迟是主要瓶颈（单次调用2-5秒）
2. 可以通过并行调用（风险识别和合规检查并行）优化
3. 引入缓存：相似条款的分析结果可以缓存复用
4. 批量处理：多份合同可以并行审查
