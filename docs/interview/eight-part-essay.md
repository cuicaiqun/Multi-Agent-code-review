# 八股文题库 — 多Agent合同审查系统

## 一、多Agent系统核心概念

### Q1: 多Agent系统的编排模式有哪些？各自适用场景？

**三种主流模式：**

1. **Supervisor（中心化控制）**
   - 一个主控Agent调度多个子Agent
   - 优点：控制精确，便于监控
   - 缺点：主控Agent是单点瓶颈
   - 适用：复杂决策需要统一协调的场景

2. **Sequential（顺序流水线）**
   - Agent按固定顺序依次执行
   - 优点：简单可靠，数据流清晰
   - 缺点：灵活性低，不支持动态路由
   - 适用：有明确先后依赖的任务（如本项目）

3. **Peer-to-Peer（对等协作）**
   - Agent之间直接通信，无中心节点
   - 优点：灵活，可扩展
   - 缺点：协调复杂，难以调试
   - 适用：需要多角度讨论的开放性任务

4. **Hierarchical（层级式）**
   - 多层级Agent树，上级分配任务给下级
   - 优点：可处理大规模任务分解
   - 缺点：层级多延迟高

**本项目选择Sequential的原因**：合同审查是一个有明确先后依赖的流程，条款提取是后续所有分析的基础。

### Q2: LangGraph的状态管理机制是什么？

**核心概念：**

1. **StateGraph**：有向图，节点是函数，边是转移条件
2. **State**：共享状态对象（Pydantic BaseModel），所有节点读写同一个State
3. **Annotation**：定义State字段的合并策略（如`add_messages`追加而非覆盖）
4. **Checkpoint**：状态的序列化快照，用于持久化和恢复
5. **Conditional Edge**：根据State的值动态选择下一个节点

**与其他方案的对比：**
- vs CrewAI：LangGraph更底层、更灵活；CrewAI更高层、更快上手
- vs AutoGen：LangGraph是图结构；AutoGen是对话循环
- vs 自己实现状态机：LangGraph提供checkpoint、interrupt、重放等企业级特性

### Q3: LangGraph的interrupt/resume机制怎么实现的？

```python
# 在构建图时设置中断节点
graph.add_node("human_review", human_review_node)

# 编译时设置checkpoint
checkpointer = MemorySaver()  # 生产环境用SqliteSaver或RedisSaver
app = graph.compile(checkpointer=checkpointer, interrupt_before=["human_review"])

# 执行到interrupt点自动暂停
result = app.invoke(state, config={"configurable": {"thread_id": "xxx"}})

# 外部提交反馈后恢复执行
app.update_state(config, {"human_feedback": feedback})
result = app.invoke(None, config)  # 从中断点继续
```

**关键原理**：
1. checkpoint保存了图执行到中断点时的完整状态
2. resume时从checkpoint恢复状态，继续执行后续节点
3. thread_id确保同一个会话可以恢复

## 二、大模型与Prompt Engineering

### Q4: 如何保证Agent输出的稳定性？

**五个策略：**

1. **Schema约束**：要求输出严格JSON格式，定义清晰的字段和枚举值
2. **低温度采样**：temperature=0.1，减少随机性
3. **分步提示(CoT)**：将复杂任务分解为步骤，逐步推理
4. **Few-shot示例**：在Prompt中提供输入输出示例
5. **输出验证**：解析输出并校验是否符合预期Schema，不符则重试

### Q5: 为什么需要三层校验架构？

| 层级 | 技术 | 确定性 | 延迟 | 适用场景 |
|------|------|-------|------|---------|
| 第一层 | 规则引擎 | 100% | <5ms | 格式校验、必要条款检查 |
| 第二层 | Agent编排 | ~90% | ~1s | 条款分类、风险路由 |
| 第三层 | 大模型 | ~80% | 2-5s | 语义分析、建议生成 |

**核心思想**：能用规则确定的就不用模型，降低成本、提高速度、保证确定性。

### Q6: RAG在法律文本中的应用和挑战？

**应用场景**：
1. 检索历史审查案例作为参考
2. 检索相关法律法规条文
3. 标准合同模板比对

**挑战**：
1. **法律文本的特殊性**：专业术语密集，通用Embedding模型效果差
2. **长文本**：合同可能几十页，需要chunk策略
3. **精确引用**：法律场景需要精确到条款号引用
4. **时效性**：法律法规会更新

**解决方案**：
- 使用法律领域微调的Embedding模型
- 按条款切分而非固定长度切分
- Hybrid Search（向量检索 + BM25关键词检索）

## 三、规则引擎

### Q7: 规则引擎和大模型各自的优劣？

| 维度 | 规则引擎 | 大模型 |
|------|---------|-------|
| 确定性 | 100% | 80-90% |
| 速度 | 极快(<5ms) | 慢(2-5s) |
| 成本 | 几乎为零 | 按token计费 |
| 灵活性 | 低（需编程） | 高（自然语言） |
| 可解释性 | 完全可解释 | 黑盒 |
| 语义理解 | 不支持 | 强 |
| 维护成本 | 需要规则管理 | 需要Prompt管理 |

**最佳实践**：两者结合，规则处理确定性场景，大模型处理语义场景。

### Q8: Drools的Rete算法是什么原理？

**Rete算法核心**：
1. 将规则编译成一棵共享的节点网络
2. 事实(Fact)从根节点进入，沿匹配路径传播
3. Alpha节点：单条件测试（如 `age > 18`）
4. Beta节点：多条件关联（如 `person.city == order.city`）
5. 终端节点：规则完全匹配，触发动作

**优势**：避免重复评估，事实变更时只需增量更新，适合大量规则的场景。

**本项目中**：规则数量不多（<100条），使用顺序匹配即可满足性能要求。Drools更适合规则数量超过1000条的企业级场景。

## 四、架构设计

### Q9: 向量数据库选型对比？

| 数据库 | 语言 | 适用场景 | 性能 |
|--------|------|---------|------|
| FAISS | C++/Python | 本地开发、中小规模 | 极快(内存) |
| ChromaDB | Python | 快速原型、嵌入式 | 快 |
| Milvus | Go/C++ | 大规模生产环境 | 高(分布式) |
| Pinecone | 云服务 | 托管服务、无运维 | 高 |
| pgvector | SQL | 已有PostgreSQL | 中等 |

**本项目选择FAISS的原因**：
1. 轻量级，pip install即可使用
2. 内存索引，检索速度极快
3. 合同审查场景数据量不大（几万条）
4. 无需额外运维

### Q10: 人机协同的设计模式？

**三种模式**：

1. **Pre-review（预审模式）**：AI先审，人后审确认 ← 本项目使用
2. **Co-review（协同模式）**：AI和人同时审查，对比结果
3. **Post-review（后审模式）**：人先审，AI辅助查漏补缺

**设计要点**：
- 明确中断点：哪些情况需要人工介入
- 超时机制：人工审核超时后的处理策略
- 审计日志：所有人工操作可追溯
- 反馈闭环：人工反馈用于持续优化AI

## 五、Java/Go特有八股

### Q11: Spring AI和LangChain的对比？

| 维度 | Spring AI | LangChain (Python) |
|------|----------|-------------------|
| 语言 | Java | Python |
| 生态 | Spring Boot全家桶 | Python AI生态 |
| 模型支持 | OpenAI/Anthropic等 | 更多模型适配 |
| Agent | 需要自行编排 | LangGraph原生支持 |
| 企业特性 | 天然的DI/AOP/事务 | 需要额外集成 |
| 社区 | 快速增长 | 非常成熟 |

### Q12: Go的Eino框架和LangChain的对比？

- Eino是字节跳动开源的Go语言LLM框架，灵感来自LangChain
- Go的优势：编译型语言、高并发性能、容器化友好
- 支持Chain/Graph/Workflow三种编排方式
- 内置DeepAgent支持复杂的多步骤任务

### Q13: 多Agent间的上下文传递策略？

1. **共享状态模式**（本项目使用）：所有Agent读写同一个State对象
2. **消息传递模式**：Agent间通过消息队列通信
3. **黑板模式**：Agent读写共享的"黑板"数据结构
4. **事件驱动模式**：Agent发布/订阅事件

**选择共享状态的原因**：
- LangGraph原生支持
- 简单直观
- 便于checkpoint和恢复
- 适合有明确数据流向的流水线场景

## 六、系统设计

### Q14: 如何设计高可用的合同审查系统？

1. **无状态服务**：API层无状态，可水平扩展
2. **异步处理**：长时间审查任务放入消息队列
3. **重试机制**：LLM调用失败自动重试（指数退避）
4. **熔断器**：LLM连续失败时切换为纯规则方案
5. **Checkpoint**：流水线可从任意步骤恢复
6. **多模型路由**：主模型不可用时切换备用模型

### Q15: 合同审查系统的性能优化策略？

1. **并行执行**：风险识别和合规检查可并行（无依赖）
2. **缓存**：相似条款的分析结果缓存复用
3. **流式输出**：大模型支持SSE流式返回
4. **批量处理**：多份合同并行审查
5. **Prompt缓存**：MiniMax支持Prompt Caching，降低90%的token成本
6. **异步IO**：使用asyncio/协程避免IO阻塞
