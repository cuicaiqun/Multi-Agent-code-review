# 架构设计文档

## 1. 系统总体架构

本系统采用**四Agent流水线 + 人机协同**的架构模式，参考了业界主流的企业级多Agent系统设计。

### 1.1 架构分层

```
┌─────────────────────────────────────────────┐
│              表现层 (Presentation)            │
│         Next.js 前端 + REST API              │
├─────────────────────────────────────────────┤
│              编排层 (Orchestration)           │
│    LangGraph / Spring AI Graph / Eino ADK   │
├─────────────────────────────────────────────┤
│              Agent层 (Agent)                 │
│  条款提取 │ 风险识别 │ 合规检查 │ 修改建议     │
├─────────────────────────────────────────────┤
│              能力层 (Capability)              │
│   法律NLP │ 规则引擎 │ 大模型API │ 向量检索    │
├─────────────────────────────────────────────┤
│              数据层 (Data)                    │
│  PostgreSQL │ 向量数据库 │ 对象存储 │ 缓存     │
└─────────────────────────────────────────────┘
```

### 1.2 核心设计原则

1. **确定性优先**：规则引擎处理可确定的检查项，大模型处理需要语义理解的场景
2. **三层校验**：规则硬校验 → Agent编排调度 → 大模型精审
3. **人机协同**：高风险条款必须经过人工确认
4. **可观测性**：全链路日志、OpenTelemetry追踪
5. **多租户隔离**：数据、模型调用、配额独立

## 2. Agent详细设计

### 2.1 条款提取Agent (Clause Extraction Agent)

**输入**：合同文档（PDF/DOCX）
**输出**：结构化条款列表

处理流程：
1. 文档解析：PDF/DOCX → 纯文本
2. 条款分割：基于标题层级和语义边界切分
3. 实体识别：识别甲乙方、金额、日期、标的物等
4. 条款分类：将条款归类（付款、违约、保密、知识产权等）

### 2.2 风险识别Agent (Risk Identification Agent)

**输入**：结构化条款列表
**输出**：风险评估报告

处理流程：
1. 规则预筛：基于规则引擎快速识别明显风险
2. 语义分析：调用大模型进行深度语义理解
3. 偏向分析：识别条款对买方/卖方的偏向程度
4. 风险评分：综合评分（高/中/低），附带详细说明

### 2.3 合规检查Agent (Compliance Check Agent)

**输入**：结构化条款列表 + 风险评估
**输出**：合规检查报告

处理流程：
1. 法规库匹配：检查是否违反相关法律法规
2. 模板比对：与标准合同模板进行条款级比对
3. 缺失检测：检查是否缺少必要条款
4. 合规评分：输出合规状态和改进建议

### 2.4 修改建议Agent (Suggestion Agent)

**输入**：风险评估 + 合规检查报告
**输出**：修改建议文档

处理流程：
1. 建议生成：针对每个风险点生成修改建议
2. 版本对比：生成条款级语义diff
3. Track Changes：输出Word Track Changes格式
4. 优先级排序：按风险等级排序建议

## 3. 三层校验架构

```
第一层：规则硬校验（确定性 100%）
├── 金额格式校验
├── 日期逻辑校验
├── 必要条款存在性检查
└── 法规红线检查

第二层：Agent编排调度（确定性 ~90%）
├── 条款分类与路由
├── 风险等级初判
├── 合规状态评估
└── 结果一致性校验

第三层：大模型精审（确定性 ~80%）
├── 语义歧义分析
├── 上下文关联分析
├── 修改建议生成
└── 自然语言解释
```

## 4. 人机协同机制

### 4.1 中断点设计

- **风险评估后**：高风险条款需人工确认
- **合规检查后**：合规异常需法务人员审核
- **建议生成后**：修改建议需业务方确认

### 4.2 协同流程

1. Agent完成自动审查，标记需人工处理的项目
2. 通知相关人员进行审核
3. 人工审核结果反馈给Agent
4. Agent根据反馈调整后续处理

| 生态成熟度 | 最成熟 | 快速发展 | 新兴 |
